Why Kaichi exists

This is how software is really shipped.

When you're shipping to a billion users, software isn't vibes. It's a brutal, unglamorous process most people never see: requirements pinned down, architecture argued out and recorded, designs torn apart before anyone writes a line, code reviewed, tested, staged, and signed off. That process — the Software Development Lifecycle — is the only reason a stack that size doesn't fall over.

Vibe coding throws all of it out: generate something plausible, eyeball it, ship it, hope. Fine for a weekend toy. A liability at scale.

Kaichi puts the lifecycle back. It does the real engineering — requirements, design, build, test, review — and leaves the documented trail behind every decision. You can hand the work to AI and still answer for it: the diff survives review, the docs are there, and every claim is checked against a gate before it advances.

It's the AI harness you can actually bring to work — code that holds up, documentation that's real, and a trail you can stand behind.

Platform support — Linux
Supported
Ubuntu 20.04+, Debian 10+, RHEL 8+ (and equivalents)
glibc ≥ 2.28 · x86_64 & arm64
Unsupported
Alpine Linux / any musl-libc distro
no glibc — the binary cannot start

Alpine/musl? The binary is dynamically linked against glibc (≥ 2.28), so it can't start on musl-only systems. Workaround (unofficial): install a glibc shim (apk add gcompat) or run inside a glibc container (e.g. debian:12-slim).
