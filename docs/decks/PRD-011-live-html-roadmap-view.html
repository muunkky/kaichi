PRD-011 · kaichi · v1.2/m2/html-explorer · Draft

The HTML roadmap should be a live dashboard, not a snapshot you regenerate by hand

PRD-006 delivered the interactive HTML roadmap view. This PRD fixes the workflow around it: a stable URL you can bookmark, kept current automatically as you edit.

PRD-011 · d56a0419 · 1/9

The problem

Every roadmap edit strands the browser tab — the file is renamed and never re-rendered

  • Each render_roadmap(format="html") writes a date-stamped file (roadmap-2026-03-28.html), so a bookmarked tab points at yesterday's file — you have to find and open the new one every time.
  • upsert_roadmap and delete_from_roadmap edit the YAML and return success, but have no awareness of the HTML view — you must remember to re-render, by hand, after every edit.
  • So the view becomes a point-in-time snapshot you generate once and forget, rather than a live dashboard you keep open while working.

The gap isn't the view — PRD-006 delivered that. It's the workflow around it: the view needs a stable URL and a render pipeline that stays current without manual intervention.

PRD-011 · d56a0419 · 2/9

Why now

The render pipeline already works — only the last-mile workflow is missing

PRD-006 and v1.2/m2 delivered the explorer, and the rendering pipeline is solid: Jinja2 templates produce self-contained HTML with inlined CSS/JS, snapshot extraction computes progress from leaf statuses, and the interactive tree with localStorage persistence works well.

What's missing is the last-mile developer experience — keeping the view alive during active editing sessions.

This is low-hanging fruit. The core rendering infrastructure exists; the changes are small in scope but high in daily-use impact.

PRD-011 · d56a0419 · 5/9

Why it's cheap

Re-rendering the whole view on each edit is invisibly cheap — the simple path wins

<50ms Full re-render after an edit (100-node roadmap): extract + Jinja2 + write

The PRD weighs three ways to keep a browser view current without a running server. The simplest is also the fastest to reason about and carries zero architectural risk.

Option A — re-render Jinja2 each edit

Chosen. Proven pipeline, ~single-digit ms render, imperceptible cost. Simplest, zero risk.

Option B — static shell + separate JSON

Blocked: file:// bans fetch() in all modern browsers. Needs a server or inline JSON anyway — negating the benefit.

Option C — inline JSON, surgical replace

Over-engineering: string-surgery complexity to save milliseconds on an operation already under 50ms.

PRD-011 · d56a0419 · 6/9

Scope boundary

Stable filename plus opt-out auto-render — nothing that needs a running server

In scope

  • write_html_view() writes a stable roadmap.html instead of a date-stamped name
  • render_html: bool = True parameter on upsert_roadmap and delete_from_roadmap
  • Auto-render logic: extract snapshot → render HTML → write to the stable path
  • html_view and non-fatal html_view_warning fields in tool responses

Out of scope — deliberate non-goals

  • Live reload / WebSocket push — requires a running server; users refresh manually
  • Client-side rendering migration — the server-render path is fast enough
  • Board HTML view auto-render — same pattern, separate scope, later
  • Watching / polling the YAML for changes — over-engineering; tool-triggered render suffices
PRD-011 · d56a0419 · 7/9

What's settled

No decisions are open — one phase, with ADR-039's date-stamped convention superseded

  • One delivery phase, no decisions needed — a straightforward extension of existing patterns, with no new dependencies.
  • One design call recorded: move _write_html_view into kaichi/views/html.py as a public function, so roadmap tools import the full render pipeline without a circular import — and the board view can reuse it later.
  • Convenience, not a transaction: if auto-render fails, the YAML edit still succeeds and the response carries a non-fatal html_view_warning. Old date-stamped files are left in place, harmless.
  • Deferred by design: board-view adoption, scoped auto-renders, and a --watch mode — each a follow-up if demand materializes.

This PRD supersedes ADR-039's date-stamped filename convention ({subject}-{date}.html → {subject}.html); ADR-039 is amended as part of delivery.
