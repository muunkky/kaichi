CWD-Pin Git Operations · ADR-051

Stop silent wrong-branch git writes: pin every scaffold git command to the parent repo, and enforce it with a hook.

A defensive fix for a harness bug — the dispatcher's shell silently drifts into a worktree between commands. Three small parts: pinned SKILL prose, one new hook, and a single settings entry.

cwd-pin-git-operations · d56a0419

The problem

Between two commands, the agent's shell silently moves into a worktree — so the next merge or push hits the wrong branch.

The Claude Code Bash tool occasionally changes working directory between calls. A git merge, push, or checkout then operates on an executor's worktree branch instead of the parent sprint branch. What makes it dangerous is how it fails:

  • Silent — no error; the wrong-target write just succeeds and looks fine.
  • In the harness — it originates below kaichi, so it can't be fixed from inside; the durable answer is to make our own commands drift-proof.
  • Every session, on the riskiest operations — merge, push, branch delete — are exactly the commands exposed.

The approach

Pin every write-class git command to the parent repo, and reject unpinned ones with a default-on hook.

One expression resolves the parent repo root from anywhere — even inside a drifted worktree subdirectory — and every git command is anchored to it with -C:

PARENT="$(dirname "$(git rev-parse --path-format=absolute --git-common-dir)")"
git -C "$PARENT" merge worktree-agent-foo --no-edit

A PreToolUse: Bash hook then scans every command and rejects unpinned write-class git (exit 2, with a teaching message); read-class git is allowed. The key tradeoff: every git block gets slightly more verbose — accepted, because the alternative is a class of silent corruption.

Architecture

Three parts, only one of them new code: pinned SKILL prose, the hook, and a single settings entry that ships it.

Scaffold SKILL prose

Every write-class git block rewritten to the pinned form. Pure text edits, no logic — the template renderer is untouched.

The hook · new code

cwd-pin-check.sh reads the Bash payload, scans for git <subcmd>, allows read-class and pinned writes, rejects the rest.

Settings wiring

One hooks.PreToolUse entry so kaichi init deploys the hook to every workspace, merged idempotently.

The hook is the only new surface; everything else reuses the existing hook pattern already proven by the direct-state-edit guard.

Alternatives — how the hook reads a command

Scan the whole command with one regex, because splitting it into segments silently lets xargs git branch -D through.

Per-segment first-token check

Split on && / ; / | and check each segment's first word. Silently exempts xargs -r git branch -D — the write hides inside xargs's arguments. Rejected.

Full shell tokeniser

Spin up a shell to parse the command properly. Too heavy for a hook that must finish in milliseconds, before the command runs. Rejected.

Whole-command regex scan

Walk every git <subcmd> substring; allow read-class and -C/cd-pinned writes, reject the rest. Closes the xargs leak; the SKILLs also restructure those pipelines into explicit loops.

Right-sizing — enforcement rollout

Default-on, with a one-release advisory mode — the middle path between a backstop nobody installs and a cliff nobody survives.

Simpler

Default-off, opt-in. A backstop nobody turns on is effectively absent. Rejected.

This design

Default-on enforce, plus KAICHI_CWD_PIN_HOOK_MODE=advisory for one release (logs, doesn't reject) and an audited per-invocation bypass — then advisory is removed.

Bigger

Default-on with no migration mode. Too aggressive for downstream forks — the dispatcher just refuses to operate, with only bypass-spam as escape. Rejected.

The same "minimum that works" logic keeps the resolver inline (no helper script to keep on PATH) and enforces write-class only — read-class failures are loud, though the SKILLs still pin reads to protect read-into-write chains.

Implementation sequence

Gate first, then build, pin, and only then wire in — Phases 2 and 3 land together so kaichi never rejects itself.

  1. Phase 0 · Gate

    File the upstream Claude Code issue and record its number, so the rollback trigger has a real referent; flip ADR-051 to Accepted.

  2. Phase 1 · Build the hook

    cwd-pin-check.sh with full subprocess-driven tests — every accept/reject case plus a corpus-regression pass over real dispatcher transcripts. Not yet wired in.

  3. Phase 2 · Pin the SKILLs

    Rewrite all 17 write-class git invocations to the pinned form; a render-and-grep test keeps them pinned as SKILLs evolve.

  4. Phase 3 · Wire it in

    One settings entry ships the hook via kaichi init. Lands lockstep with Phase 2 for this repo — otherwise the dispatcher would reject its own still-unpinned commands.

  5. Phase 4 · Document

    Changelog (breaking change, migration flag, deprecation timing), a CLAUDE.md pointer, and backlog cards for removing advisory mode and revisiting after an upstream fix.

Open questions

A few classification edges left to iterate — none block the build.

  • Non-git commands: should the hook also pin CWD-sensitive commands like pytest or make? v1 is git-only; an open extension point.
  • Where the regression corpus lives: committed real transcripts (path-leak risk) versus generating them from rendered SKILLs. Leans generated.
  • Classifying git stash: treat it as write-class by the safe default, even though a drifted stash is benign per-worktree.
